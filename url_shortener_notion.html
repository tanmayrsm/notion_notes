Hash & Collision — Conceptual Deep Dive

This is the part most people get wrong in interviews. Let's be very precise.

Why MD5 / SHA-256 alone is not enough

MD5 produces a 128-bit digest — 32 hex characters. We only need 7 base62 characters (~42 bits). So we truncate. Truncation is what introduces collision risk. You're projecting a near-infinite input space into a much smaller slot space.

Collision Resolution Strategies

①  Salt + Re-hash ▶

Append a version salt to the original URL and hash again. The tiniest change to the input produces a completely different MD5 output.

# Attempt 1
new_input = url + "v1"
short_code = base62(md5(new_input)[:8])  # "mR4xK9z"

# Check DB
SELECT original_url FROM urls WHERE short_code = 'mR4xK9z';
→ 0 rows — FREE. Save it.

# If still collides, try v2, v3. Cap at 3 attempts.
# After 3 failures → fall back to ID pool.
⚠️
Gotcha: Same URL + different salts = different short codes. Breaks idempotency. Fix by maintaining a url_hash → short_code dedup index so the same long URL always returns the same short code, regardless of how many times it's submitted.
②  Pre-generated ID Pool (Bitly / Twitter approach) ▶

A background worker pre-generates millions of verified-unique codes and stocks them in a Redis list. Write service just pops one — zero collision possible at write time.

# Background worker (runs 24/7, stays ~10M codes ahead):
loop:
    code = random_base62(7)
    if not bloom_filter.check(code):   # O(1) — no DB hit
        RPUSH url_id_pool code

# Write service (on each incoming URL):
code = LPOP url_id_pool            # atomic pop
→ Guaranteed unique. No DB check. No retry loop.
INSERT INTO urls (code, original_url, ...)
③  Snowflake ID + Base62 Encode ▶

Use a distributed globally-unique monotonic counter (Snowflake format), then base62-encode it into your short code.

# Snowflake ID structure (64 bits):
bit 63     : unused (sign)
bits 22-62 : timestamp_ms - epoch   (41 bits = 69 years)
bits 12-21 : machine_id              (10 bits = 1024 nodes)
bits 0-11  : sequence_per_ms         (12 bits = 4096/ms)

id = (ts - epoch) << 22 | machine_id << 12 | sequence
short_code = base62_encode(id)        # "xK7mP2q"

# Pros: no collision, sortable by time
# Cons: sequential = enumerable. XOR-mask before encoding to obscure.
④  Bloom Filter Guard ▶

A Bloom filter is a probabilistic structure: false positives possible, false negatives impossible. Used as a fast pre-check before hitting the DB during pool generation.

# Bloom filter properties at 1 billion entries, 0.1% FP rate:
Memory usage  : ~1.8 GB  (fits in Redis)
Check latency : O(1)
False negative: IMPOSSIBLE  → if says "not exists", guaranteed safe
False positive: ~0.1%       → if says "exists", do a real DB lookup

# Result: 99.9% of uniqueness checks need ZERO DB reads.
# Only 0.1% false positives trigger a real SELECT.
✅
Recommended at Uber scale: Pre-generated ID Pool + Bloom Filter guard. The pool eliminates collision at write time. The Bloom filter makes pool generation fast. Custom aliases bypass the pool and are validated via DB unique constraint.

Concrete Example — URL 1 (Happy Path)

🔗
Input: https://example.com/abc/daf?user=123&session=xyz
1
Run MD5 on the full URL string
MD5 always returns exactly 32 hex characters (128 bits). Two URLs differing by even one character give completely different digests.
✓
Result
MD5("https://example.com/abc/daf?user=123&session=xyz")
= a3f8c2d1e94b76052f1830c7d6a49e81
  ↑ first 8 chars → use these only
2
Convert first 8 hex chars → decimal → base62
Take the first 8 hex chars, interpret as a 32-bit unsigned integer, then encode in base62 (digits 0–9, a–z, A–Z).
✓
First 8 hex : a3f8c2d1
Hex → dec   : 0xa3f8c2d1 = 2,751,546,065

Base62 alphabet: 0123456789abcdefghijklmnopqrstuvwxyzABCDEFGHIJKLMNOPQRSTUVWXYZ

2,751,546,065 mod 62 repeatedly:
  remainder 3  → '3'
  remainder 25 → 'p'
  remainder 17 → 'h'
  remainder 28 → 's'
  remainder 7  → '7'
  remainder 52 → 'Q'
  remainder 11 → 'b'

Short code = 3phs7Qb
3
Check DB — is this code already taken?
SELECT original_url FROM urls WHERE short_code = '3phs7Qb';
→ 0 rows. Code is free. No collision.
4
Save to DB + warm Redis cache
INSERT INTO urls (short_code, original_url, created_at)
VALUES ('3phs7Qb', 'https://example.com/abc/daf?user=123&session=xyz', NOW());

# Redis cache warm-up
SET url:3phs7Qb "https://example.com/abc/daf?user=123&session=xyz" EX 86400
✓
Return short URL to user
https://short.ly/3phs7Qb
Total write time: ~30ms. Done.

ID Pool Method (Industrial Standard)

The hash+salt approach works but has non-deterministic latency under load. The ID pool eliminates collision entirely at write time. Used by Bitly and Twitter.

# Background worker — fills the pool continuously:
while True:
    code = random_base62(7)
    if bloom_filter.check(code) == "definitely_not_exist":
        RPUSH url_id_pool code         # safe — no DB lookup needed
    elif bloom_filter.check(code) == "might_exist":
        SELECT 1 FROM urls WHERE short_code = code
        if not_found:
            RPUSH url_id_pool code      # bloom was a false positive, still safe

# Write service — on each incoming URL:
# ─────────────────────────────────────
code = LPOP url_id_pool           # atomic, O(1)
→ Guaranteed unique. No hash. No collision check. No retry.
INSERT INTO urls (code, original_url, NOW())

# Pool monitoring:
pool_size = LLEN url_id_pool
if pool_size < 1_000_000:  alert("Pool low!")
if pool_size < 100_000:   switch_to_inline_hash_fallback()
if pool_size == 0:        return HTTP 503, Retry-After: 30

System Architecture

Write Path — Step by Step

1
POST /api/shorten { url, custom_alias?, ttl? }
API Gateway validates JWT, enforces rate limit (e.g. 100 req/min/user), rejects URLs > 8192 bytes with 414.
2
Deduplication check
Compute SHA256(url), look up url_hash → short_code in dedup index (cached in Redis). If found → return existing code immediately. Idempotent.
3
Acquire short code
Custom alias → validate uniqueness in DB (unique constraint). Auto-generate → LPOP url_id_pool from Redis (atomic). If pool empty → inline hash+salt fallback.
4
Write to primary DB
Insert into urls table and url_hash_index dedup table. DB enforces unique constraint on short_code as final safety net.
✓
Warm Redis cache + return 201
Write url:{code} → original_url into Redis with TTL. Return short URL to client. p99 target: <50ms.

1

CDN checks edge cache. Hot URLs (top 20%) respond in ~2ms — no origin hit. Returns 302 with Location header.
2
CDN miss → Redis lookup
GET url:3phs7Qb from Redis. Hit rate target >99%. Response in microseconds.
3
Redis miss → Cassandra read
Primary key lookup on short_code. Check expires_at — if expired return 410 Gone, if not found return 404. Repopulate Redis on hit.

301 vs 302 — The Decision That Matters at Scale

🔒 301 Permanent Redirect
Browser caches it permanently. Second click from same browser never hits your servers. Massively reduces origin traffic. Cost: you lose analytics after the first click — browser doesn't re-request, so you can't count impressions. Use for internal tools where analytics don't matter.
🔄 302 Temporary Redirect
Every single click goes to your servers. Full click-by-click analytics — required if you're billing advertisers. Cost: At 100B redirects/day this is why the CDN layer is non-negotiable. Without CDN you'd need hundreds of thousands of servers just for redirects.
💡
Interviewer signal: Choosing 301 vs 302 is a product decision disguised as a technical one. The "right" answer depends on the business model. Advertising-driven → 302. Cost-optimisation-first → 301 + CDN-level analytics.

Edge Cases — Every One Must Be Handled

🚫 Malicious / Phishing URLs
Validate URL format on write. Integrate Google Safe Browsing API asynchronously. Store is_malicious flag. On redirect, if flagged → 451 Unavailable For Legal Reasons + warning page. Periodic re-scan via background worker.
⏰ URL Expiry — Soft vs Hard Delete
Soft delete: set is_active = false. Hard delete: background cron sweeps expired rows after grace period. Expired URL returns 410 Gone (not 404 — tells crawlers it was intentional). Purge from Redis immediately on expiry detection.
📉 ID Pool Exhaustion
Three-level degradation: (1) Alert fires at <1M codes, (2) fallback to inline hash+salt at <100K codes, (3) HTTP 503 + Retry-After at 0. Worker runs with leader election (ZooKeeper/etcd) — if it dies, follower takes over in 30s.
🌊 Cache Stampede on Viral URL
When a viral URL's cache entry expires, thousands of requests simultaneously miss and hammer the DB. Fix: probabilistic early expiration (XFetch algorithm) + request coalescing (Redis SETNX mutex — only one request fetches from DB, others wait on pub/sub).
🔀 Custom Alias Conflicts
Custom aliases share the same table as generated codes. DB unique constraint enforced at write time. If user's chosen alias collides with an existing code → immediate 409 Conflict. No namespace separation needed — uniqueness constraint handles it.
♻️ Same URL Submitted Twice
Store SHA256(original_url) → short_code in a dedup index table. Before generating a new code, check this index. If found → return existing code. Makes the write API fully idempotent. Cache this lookup in Redis.
🌍 Cross-Region Consistency
Writes go to primary region (strong consistency). Reads from local replica (eventual, ~100ms lag). Acceptable — a newly created URL may 404 for a few seconds in distant regions. Mitigate with synchronous replication to 2 regions for critical writes.
💥 DDoS on Redirect Endpoint
Per-IP rate limiting at CDN layer (token bucket). If a specific short code gets hammered (>10K req/sec): circuit breaker — rate limit at code level, serve cached response only, throttle Kafka analytics writes. DB writes stop; CDN absorbs the flood.

DB Schema

-- Primary table (Cassandra / DynamoDB)
-- Partition key is short_code → O(1) point lookup
urls (
  short_code    TEXT    PRIMARY KEY,     -- e.g. "3phs7Qb"
  original_url  TEXT    NOT NULL,
  user_id       UUID,
  created_at    TIMESTAMP,
  expires_at    TIMESTAMP,                -- NULL = never expires
  is_active     BOOLEAN DEFAULT TRUE,   -- soft delete flag
  click_count   COUNTER                   -- CRDT, no race condition
);

-- Dedup index (Redis hash or separate Cassandra table)
url_hash_index (
  url_sha256    TEXT    PRIMARY KEY,     -- SHA256(original_url)
  short_code    TEXT                      -- maps back to urls table
);

-- Pre-generated ID pool (Redis list)
-- Background worker: RPUSH url_id_pool "xK7mP2q"
-- Write service:     LPOP  url_id_pool  (atomic pop)

-- Why Cassandra over MySQL?
-- • Workload = pure point lookups by PK (short_code). No JOINs.
-- • Hash-partitioned ring → O(1) reads, linear write scaling.
-- • Native active-active multi-region replication.
-- • COUNTER type for click_count → no lock contention.
-- • At >99% cache hit rate, DB sees ~1M reads/day — Cassandra handles trivially.
Design ChoiceWhy
No auto-increment PKPrevents sequential ID enumeration attack — attacker can't scrape all URLs by incrementing
COUNTER for clicksCassandra's native CRDT counter — concurrent increments without race conditions
expires_at indexedBackground TTL worker uses this for batch expiry sweeps
Separate dedup tableKeeps primary table's partition key clean — no secondary index overhead on hot path
is_active flagSoft delete preserves audit trail; hard delete happens offline in batch, never on the read path
3 AZ replicasSurvive full AZ outage with zero data loss. Reads from nearest replica reduce cross-AZ latency costs
🧠
The mental model in one paragraph:

A URL shortener is a glorified key-value store with a globally unique ID generation problem. The hard parts are: (1) generating short codes without collision at massive scale — solved by pre-generated ID pool + Bloom filter, (2) making reads sub-10ms at 100B/day — solved by CDN → Redis → DB layering with >99% cache hit, and (3) keeping the write path idempotent — solved by SHA256-based dedup index. Every other complexity (analytics, expiry, malicious URL detection) lives off the hot path so it never touches redirect latency.
