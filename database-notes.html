🐘
PostgreSQL

The one you already know. Stores data in rows, each row containing all column values together. The query planner handles everything — you describe what you want, Postgres figures out how to get it.

CREATE TABLE colleges (
  college_id   SERIAL PRIMARY KEY,
  college_name VARCHAR(200),
  city         VARCHAR(100)
);

CREATE TABLE users (
  user_id    SERIAL PRIMARY KEY,
  name       VARCHAR(100),
  email      VARCHAR(200),
  college_id INT REFERENCES colleges(college_id) -- FK enforced
);
SELECT u.name, u.email, c.college_name, c.city
FROM   users    u
JOIN   colleges c ON u.college_id = c.college_id;

-- Postgres does the matching internally.
-- You can JOIN any tables, any columns, any time.
CREATE VIEW user_with_college AS
  SELECT u.user_id, u.name, u.email, c.college_name, c.city
  FROM  users u JOIN colleges c ON u.college_id = c.college_id;

-- Query it exactly like a table:
SELECT * FROM user_with_college WHERE city = 'Hyderabad';
✅
A VIEW is just a saved query — no extra storage, always reflects live data.
CREATE MATERIALIZED VIEW user_college_fast AS
  SELECT u.name, u.email, c.college_name
  FROM  users u JOIN colleges c ON u.college_id = c.college_id;

-- Result is stored on disk → very fast to query.
-- Must refresh manually when data changes:
REFRESH MATERIALIZED VIEW user_college_fast;
⚠️
Use materialized views when joins are slow and data doesn't change every second (e.g. end-of-day reports).
💠
Apache Cassandra
Apache Cassandra — originally built at Facebook, now an Apache project

Built for massive write workloads across many machines. The radical tradeoff: no joins, ever. Instead, you design your tables around your queries.

🔑
The Cassandra mindset flip: In Postgres you normalize first, query however you want. In Cassandra you decide your queries first, then design your table around them.
-- One table shaped for "get all users by college"
CREATE TABLE users_by_college (
  college_name  TEXT,
  user_id       UUID,
  user_name     TEXT,
  email         TEXT,
  city          TEXT,       -- college info embedded here
  PRIMARY KEY (college_name, user_id)
  --          ↑ partition    ↑ clustering
);
-- FAST — queries by partition key
SELECT user_name, email FROM users_by_college
WHERE college_name = 'IIT Hyderabad';

-- SLOW / BAD — no partition key
SELECT * FROM users_by_college WHERE city = 'Hyderabad'; -- ❌ full scan
-- Same data, different primary key = second access pattern
CREATE MATERIALIZED VIEW users_by_id AS
  SELECT * FROM users_by_college
  WHERE college_name IS NOT NULL AND user_id IS NOT NULL
  PRIMARY KEY (user_id, college_name);
⚠️
Cassandra MVs are a second physical copy auto-synced on writes. Many teams skip them and maintain two tables manually — MVs have reliability issues in production.
☁️
DynamoDB

Same philosophy as Cassandra (design for your queries, no joins) but fully managed by AWS. You never run a server — just define your table, set capacity, and pay per read/write.

▶ Single-table design — what it looks like 
// Users and colleges in ONE table, prefixed keys
{ "PK": "COLLEGE#iit-hyd", "SK": "METADATA", "college_name": "IIT Hyderabad" }
{ "PK": "USER#arjun",      "SK": "PROFILE",  "name": "Arjun", "college_id": "iit-hyd" }
▶ GSI — Global Secondary Index (DynamoDB's answer to views)

A GSI is a separate automatically-maintained copy of your table with a different key. It's how you support a second access pattern without duplicating data manually.

// "ByCollege" GSI lets you query all users for a college
const result = await client.send(new QueryCommand({
  TableName: "AppTable",
  IndexName: "ByCollege",
  KeyConditionExpression: "college_id = :cid",
  ExpressionAttributeValues: { ":cid": { S: "iit-hyd" } }
}));
▶ The update problem — denormalization's catch
⚠️
When a college renames itself: Postgres = 1 UPDATE. DynamoDB = update every user row that has the college name embedded. At 10,000 users, that's 10,000 writes.
🪨
RocksDB (in Apache Flink)
RocksDB uses an LSM-tree (Log Structured Merge tree) — optimised for write-heavy workloads by batching and merging writes

RocksDB is not a server you connect to. It's a library your application links against. There's no port, no protocol — it just reads and writes to local disk extremely fast.

In Flink, you set it as the state backend and then forget about it. Flink reads/writes RocksDB automatically whenever your job uses keyed state.

✅
With the default HashMapStateBackend, your Flink state is limited by JVM heap (RAM). Switching to EmbeddedRocksDBStateBackend lets state spill to local SSD — enabling terabytes of state per task manager.
// All you do in your Flink job:
StreamExecutionEnvironment env = StreamExecutionEnvironment.getExecutionEnvironment();
env.setStateBackend(new EmbeddedRocksDBStateBackend());

// Now all ValueState / MapState / ListState lives on disk:
ValueState<Long> count = getRuntimeContext().getState(
  new ValueStateDescriptor<>("click-count", Long.class));
public class UserCollegeJoin extends RichCoFlatMapFunction<User, College, String> {
  private ValueState<User>    userState;    // stored in RocksDB
  private ValueState<College> collegeState; // stored in RocksDB

  public void flatMap1(User u, Collector out) {
    userState.update(u);
    if (collegeState.value() != null)
      out.collect(u.name + " | " + collegeState.value().name);
  }
}
-- Flink SQL temporal join — looks just like Postgres!
SELECT u.name, u.email, c.college_name
FROM   users u
JOIN   colleges c FOR SYSTEM_TIME AS OF u.proc_time
  ON   u.college_id = c.college_id;

-- And views work too:
CREATE VIEW user_with_college AS
  SELECT u.name, c.college_name
  FROM  users u
  JOIN  colleges c FOR SYSTEM_TIME AS OF u.proc_time
    ON  u.college_id = c.college_id;

Apache Druid
Apache Druid — sub-second OLAP queries on billions of rows. Used by Netflix, Airbnb, Flipkart.

Druid is an analytical database — built for dashboards, not app data. Three things make it different from everything else:

1. Time is mandatory
Every table needs a __time column. Data is physically partitioned by time on disk.
2. Columnar storage
Each column stored separately on disk. SELECT SUM(x) only reads the x file — all other columns skipped.
3. Rollup at ingest
1 billion raw events → pre-aggregated to 500 rows at ingest time. Queries hit the tiny pre-cooked result.
📌
Druid is not a replacement for Postgres. It cannot do the things Postgres does (complex joins, app writes, ACID). It does one thing: analytics on time-series data, blazing fast.
REPLACE INTO "enrollments" USING TABLE(EXTERN(...))
SELECT
  TIME_PARSE("timestamp") AS "__time",  -- always required
  "user_id",
  "college_name",
  "city",
  "enrollments"
FROM SOURCE
PARTITIONED BY MONTH; -- Jan data separate from Feb data on disk
-- Instead of storing 1B raw events, pre-aggregate:
SELECT
  FLOOR(__time TO MONTH) AS "__time",
  "college_name",
  "city",
  COUNT(*)                AS "enrollments",
  COUNT(DISTINCT user_id) AS "unique_users"
FROM SOURCE
GROUP BY 1, 2, 3
PARTITIONED BY MONTH;
🗜️
1 billion events + 500 college×month combos → stored as 500 rows. Dashboard queries become instant.
-- Enrollments per college per month
SELECT TIME_FLOOR(__time, 'P1M') AS month,
       college_name,
       SUM(enrollments) AS total
FROM   enrollments
WHERE  __time BETWEEN '2024-01-01' AND '2024-12-31'
GROUP BY 1, 2
ORDER BY month;

-- Druid-specific: approximate distinct count (much faster)
SELECT college_name,
       APPROX_COUNT_DISTINCT(user_id) AS approx_users
FROM   enrollments
GROUP BY college_name;

Segments — how Druid partitions by time

Druid splits data into segments. Each segment is one time chunk (e.g. one month). A query with WHERE __time >= '2024-01-01' only opens relevant segments.

JOINs & VIEWs — what each database can do
Database JOIN VIEW How to "join" instead
PostgreSQL ✅ Any JOIN, any table ✅ Regular + materialized Just use a JOIN
Cassandra ❌ Never ⚠️ Materialized view (risky) Denormalize — embed related data in the row at write time
DynamoDB ❌ Never 🔀 GSI (different key copy) Embed in item, or fetch separately + join in app code
RocksDB/Flink ✅ Via Flink stream join ✅ Flink SQL CREATE VIEW CoFlatMapFunction — store both sides in state, emit when both arrive
Druid ⚠️ Small lookup table only ✅ Query alias (no storage) Embed at ingestion time, or use a Druid lookup
