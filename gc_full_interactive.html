1. What GC does

Every new SomeObject() puts an object on the heap. When nothing references it anymore, it's garbage — but sits in memory until GC removes it. The challenge: inspecting the heap safely while your app is running at full speed.

5. G1 GC — heap regions & remembered sets

G1 = Garbage First — divides the heap into equal regions and collects the most garbage-dense ones first, keeping pauses bounded by a time budget.

📦

Why regions? Traditional GC had to collect the entire old gen at once — pauses scaled with heap size. G1 can collect any subset of regions, so pause = only as much work as fits in your time budget.

Remembered sets (RSets)

If G1 only collects some regions, how does it know all objects pointing into those regions? Each region has its own RSet — a list of (other-region, card) pairs. During GC, only roots + RSets of collected regions are scanned — never the whole heap.

💸

RSet cost: RSets consume 1–5% of heap. Write-heavy workloads that update many references force background "refinement threads" to continuously update RSets, costing CPU.

6. TLAB — why object allocation is nearly free

Each thread has its own private chunk of Eden called a Thread-Local Allocation Buffer. Allocating inside = just bumping a pointer. No locks, no coordination.

7. Write barrier — keeping concurrent marking correct

Problem: your app mutates references while GC is mid-marking. A black object could get a new reference to a white (unscanned) object — GC would then delete that white object even though it's now reachable.

8. Safepoints — the hidden latency source

Before GC can run, every app thread must reach a safepoint — a safe pause location. GC waits until the slowest thread arrives. Your total pause = GC work time + time waiting for that thread.

🚨

Thread C below has a long loop with no safepoint poll inside. Watch how GC must wait for it, even though the other threads are ready immediately. This "time-to-safepoint" is a real source of mysterious long pauses in production.

11. Commands cheatsheet

Build-time — paste into your service config

# Baseline for latency-sensitive 1B rpm service
-XX:+UseZGC                              # sub-ms pauses, JDK 17+
-Xms16g -Xmx16g                        # lock heap, prevent resize pauses
-XX:+AlwaysPreTouch                      # pre-fault pages at startup
-Xlog:gc*:file=gc.log:time:filecount=5,filesize=20m
-XX:+HeapDumpOnOutOfMemoryError
-XX:HeapDumpPath=/tmp/oom.hprof
-XX:+ExitOnOutOfMemoryError              # fail fast, let k8s restart
-XX:+UseCountedLoopSafepoints            # reduce time-to-safepoint

Runtime detection

# Real-time GC stats (1s refresh). FGC should stay near 0.
jstat -gcutil PID 1000

# Allocation flame graph (find what's creating the most objects)
./asprof -e alloc -d 30 -f alloc.html PID

# JFR deep recording on running process (no restart)
jcmd PID JFR.start name=rec duration=120s filename=out.jfr

# Heap state, native memory, active flags
jcmd PID GC.heap_info
jcmd PID VM.native_memory
jcmd PID VM.flags

Emergency — free memory in production

# Option 1: trigger GC (causes pause — drain traffic first)
jcmd PID GC.run

# Option 2: heap dump then restart (safest at 1B rpm)
jcmd PID GC.heap_dump /tmp/emergency.hprof
# → restart pod, analyze dump offline in Eclipse MAT

# Option 3: clear a cache via admin endpoint
cache.invalidateAll();

12. What to say in the interview

🎯

What they're really testing: Can you reason about GC at system scale and connect JVM internals to user-facing latency? Listing flags is not the answer.

Opening — set the stakes

Say this first
"At 1 billion requests per minute, we're allocating 50–200 GB of objects per second. A 200ms GC pause means millions of requests queuing up or timing out. The design decision isn't just which collector — it's how to architect the service so GC pressure is minimal in the first place."

Collector choice

G1 vs ZGC
"I'd default to ZGC on JDK 17+. The key difference: ZGC moves objects concurrently while requests are being served — pauses stay under a millisecond. G1 must stop the world to evacuate objects, so pause time scales with how much it moves."

Code-level design

What you'd actually change
"I'd profile with async-profiler in allocation mode first. Usually 2–3 hotspots account for 80% of allocations — boxing primitives in maps, intermediate strings in hot paths, stream chains. For network payloads, DirectByteBuffer keeps large buffers off-heap entirely. The goal: make GC boring — infrequent minor collections, near-zero old-gen pressure."

Failure scenario

If they push harder
"The failure mode I'd watch for is evacuation failure — G1 can't find free regions, falls back to full STW mark-compact, pauses for seconds. I'd prevent it: size the heap with 40–50% headroom, set InitiatingHeapOccupancyPercent=35 so concurrent marking starts early, and keep FGC count in jstat near zero as a prod signal."
They askYou say
"What's a safepoint?"A pause checkpoint every thread must reach before GC starts. Time-to-safepoint = hidden latency from the slowest thread.
"What's a write barrier?"JIT-injected code on every reference write that keeps concurrent marking correct — logs overwritten references so GC re-scans them.
"What's TLAB?"Each thread's private Eden chunk. Allocation = pointer bump, no locks. Nearly free.
"What's SATB?"G1's technique: snapshot live refs at marking start, log overwrites, re-scan them. May retain floating garbage but never deletes live objects.
"Free memory in prod?"Drain pod → jcmd GC.run → heap dump → restart. Analyze dump offline. Fix root cause in code.
"Show live GC stats?"jstat -gcutil PID 1000 — updates every second, shows Eden %, Old gen %, Full GC count.
✅

The answer that wins the room: "I'd fix the allocation hotspot in code. Flags buy you time. Code changes save you permanently. GC tuning is the last resort, not the first."
