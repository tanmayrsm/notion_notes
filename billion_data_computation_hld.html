Foundations · 01

The core problem

Two distinct problems that get confused in interviews and in code.

Real-time delivery

  • Push events to live clients now
  • Low latency — under 100ms ideal
  • WebSocket or SSE connections
  • Requires stateful connection servers
  • Example: chat message, live score update

Timed delivery

  • Fire an event at exactly t+Ns
  • Trigger webhook/microservice after delay
  • Requires a dedicated delay scheduler
  • Kafka cannot guarantee this
  • Example: reminder, SLA breach ping, retry
💡
At 1 billion users both problems appear simultaneously. Your architecture must solve them independently — the real-time path and the delay path are different components with different guarantees.

Kafka · 03

What Kafka actually is

Understanding the internals is the only way to know what Kafka can and cannot do.

Kafka is a commit log

  • Records are appended to a log on disk
  • Consumers track their own offset
  • Consumers pull at their own pace
  • Records stay on disk (configurable retention)
  • Multiple consumer groups, independent offsets

Key properties

  • Ordered within a partition
  • Durable — replication across brokers
  • Replayable — reset offset anytime
  • High-throughput — millions/sec
  • Scalable — add partitions linearly

Kafka · 04

Why Kafka cannot guarantee a 50-second delay

🚫
The core limitation: Kafka is pull-based. A consumer polls on its own schedule. There is zero mechanism in Kafka to say "deliver this message to a consumer in exactly 50 seconds." Consumers decide when they poll, not Kafka.

What Kafka cannot do

  • Schedule delivery at a wall-clock time
  • Hold a message until a timeout fires
  • Push to a webhook after N seconds
  • Guarantee max delivery latency
  • Fire a callback after exactly 50s

What Kafka is perfect for

  • High-throughput ordered ingest
  • Durable buffering between services
  • Triggering the delay scheduler
  • Fan-out to many consumer groups
  • Replay for debugging / backfill

A consumer polling every 100ms will be off by up to 100ms per poll. At 1B events with variable consumer lag, poll drift compounds. The delay guarantee must live in a dedicated scheduler — not in Kafka. Kafka's role is to reliably hand the event to the scheduler.

Delay tier · 05

Redis sorted set as a delay queue

The most common primary delay mechanism at scale. Supports arbitrary delays, sub-second precision, and up to ~100M pending jobs in RAM.

How it works

A sorted set stores job_id as the member and fire_timestamp (Unix epoch) as the score. A poller runs every second, fetching all jobs whose score ≤ now.

# Enqueue: schedule job to fire in 50 seconds
fire_at = time.time() + 50
redis.zadd("scheduled_jobs", {job_id: fire_at})

# Poller: fetch and atomically claim jobs due now
def poll():
    now = time.time()
    jobs = redis.zrangebyscore("scheduled_jobs", 0, now)
    for job_id in jobs:
        removed = redis.zrem("scheduled_jobs", job_id)
        if removed:  # won the ZREM race — we own this job
            executor.submit(process_job, job_id)
🔑
The ZREM trick: Multiple pollers can call ZRANGEBYSCORE and see the same jobs. ZREM is the claim — it's atomic, so only the first caller gets 1 back. Others get 0 and skip. This is a free first layer of deduplication before the fence even fires.

Strengths

  • Sub-second precision (~1s poll interval)
  • No max delay limit — hours, days, weeks
  • O(log N) insert, O(K) fetch
  • Easy to inspect / debug
  • ZRANGEBYSCORE + ZREM pattern is safe

Limitations

  • All pending jobs live in RAM (~100M limit)
  • Redis restart risk — needs AOF/RDB
  • Single-node bottleneck unless clustered
  • Poller is a long-polling loop

SQS delay queue

AWS-managed, zero ops, at-least-once delivery. The best fallback/overflow layer. Critical constraint: max 15 minutes.

    QueueUrl="https://sqs.us-east-1.amazonaws.com/...",
    MessageBody=job_id,

⚠️
Visibility timeout trap: If your worker takes longer than the visibility timeout to process, SQS assumes it died and redelivers the message to another worker — even if it's still running. Set visibility timeout to at least 2× your max processing time.

Chaining SQS for long delays (>15 min)

SQS's 15-minute cap means you can't directly schedule a job for 1 hour. Pattern: enqueue to SQS with 15-minute delay → consumer re-enqueues with the remaining delay. Repeat until within the window. Or just use Redis sorted set for anything over 15 minutes.

Delay tier · 07

Hierarchical timing wheel

Used inside Kafka itself, Netty, the Linux kernel, and Nginx. O(1) insert and fire regardless of pending job count. Build this only if you need sub-millisecond precision at massive scale.

The mechanism

How it works: Two concentric rings. The inner ring represents minutes (ticks once per minute). The outer ring represents seconds (ticks once per second). Jobs are stored in the bucket they'll fire in.

The red hand advances one tick per second. All jobs in the current bucket fire immediately. Jobs further out wait in their bucket.

Why O(1): Inserting a job is just bucket[fire_time % wheel_size].append(job). No sorting, no heap. Firing is bucket[current_tick].drain().

📖
Real-world usage: Kafka's internal purgatory (for in-flight produce requests), Netty's HashedWheelTimer, Linux's jiffies kernel timer, Nginx's event loop. Twitter/Uber have open-sourced timing wheel implementations for JVM services.

Real-time delivery · 09

WebSocket / SSE at 1B users

The real-time path (stream processor → live clients) is a separate problem from the delay path. It requires stateful connection servers and consistent routing.

WebSocket

  • Full-duplex persistent TCP connection
  • Best for: chat, live collaboration, gaming
  • ~50K connections per server node
  • Connection is stateful — sticky routing required
  • Example: Discord (Elixir), Slack

SSE (Server-Sent Events)

  • One-way server → client over HTTP/2
  • Best for: feeds, dashboards, notifications
  • Simpler — reuses HTTP infrastructure
  • Auto-reconnects on drop
  • Example: GitHub live feed, stock tickers

Connection server topology

Each connection server holds ~50K open WebSocket connections. For a user to receive an event, the message must reach the connection server holding their socket.

Deduplication · 11

The deduplication problem

You have Redis sorted set AND SQS both firing for the same job. Add 100 worker pods racing to process. Without a fence, events fire multiple times — guaranteed.

🚨
What happens without a fence: Redis poller fires job_id. SQS also delivers job_id. 3 pods pick them up simultaneously. Webhook fires 3 times. User gets 3 push notifications. Microservice gets 3 calls. At 1B users this is not rare — it's the default.

SQS's delivery guarantee is at-least-once — it will never under-deliver, but it may over-deliver. Redis sorted set with a shared poller is also at-least-once because ZRANGEBYSCORE is not atomic with the processing. Neither queue owns deduplication. A third atomic store must be the arbiter.

Sources of duplicates

  • Redis + SQS both firing same job
  • Multiple pods polling the same sorted set
  • SQS redeliver after visibility timeout
  • Worker crash after acquire, before ack
  • Network retry on the downstream call

What we need

  • Exactly-one execution across all pods
  • Cross-pod atomic claim mechanism
  • Idempotent downstream calls as backstop
  • Retry eligibility preserved on failure
  • Zero silent job loss
Deduplication · 12

The SET NX EX fence pattern

A single Redis atomic command is the fence. NX = only set if key doesn't exist. Across 100 pods racing on the same job, exactly one gets back 1. All others get nil.

def process_job(job_id: str, source: str):
    # Atomic fence — only one worker wins across all pods
    acquired = redis.set(
        f"dedup:{job_id}", 1,
        nx=True,
        ex=120   # TTL must exceed SQS visibility timeout
    )

    if not acquired:
        # Duplicate — drop silently, ack to remove from SQS
        sqs.delete_message(job_id)
        return

    try:
        execute_job(job_id)
        sqs.delete_message(job_id)
        delivery_log.write(job_id, "delivered")
    except Exception:
        # CRITICAL: release fence so job stays retry-eligible
        redis.delete(f"dedup:{job_id}")
        raise

Deduplication · 14

Critical failure cases

Case 1 — forgetting to DEL the key on failure

Worker acquires the fence, downstream call throws. If you don't DEL dedup:{job_id}, the job is locked until TTL expires. If TTL > retry window — job is silently dead forever.

💣
Always release the fence in the exception handler. This is the single most common implementation bug in this pattern.

Case 2 — TTL shorter than SQS visibility timeout

If fence TTL = 30s and SQS visibility timeout = 60s, the key expires. SQS redelivers. A second worker acquires the fence cleanly — double execution.

⏱
Rule: TTL ≥ max(SQS_visibility_timeout, Redis_poll_interval) + 20s buffer. For a 50s job with 60s visibility timeout → use TTL = 120s minimum.

Case 3 — worker crash after acquiring fence but before acking SQS

SQS redelivers after visibility timeout expires. New worker tries fence → gets nil (key still held) → drops it. The job is silently skipped for the TTL window.

Fix: use SQS FIFO with a message deduplication ID equal to job_id. SQS deduplicates at the queue level for 5 minutes — a second layer before the fence is even checked.

Case 4 — clock skew between workers

Redis sorted set uses time.time() from the poller's local clock. If two pods have 2-second clock drift, pod A may see a job 2 seconds before pod B — and start processing while pod B's poll hasn't fired yet, leading to ZREM races. Fix: use NTP synchronisation and add a clock skew buffer to TTL.

Reference · 16

The four defence layers

Deduplication is a stack. Any one layer failing alone is survivable. All four failing simultaneously is astronomically unlikely.

1
SQS FIFO deduplication ID
Queue-level dedup within a 5-minute window. Prevents SQS from re-enqueuing the same job_id twice. Free on FIFO queues — use it.
2
ZREM atomic claim (Redis sorted set)
Multiple pollers can see a job via ZRANGEBYSCORE — but only the first ZREM returns 1. Free first-pass dedup before the fence.
3
Redis SET NX EX fence
The real guarantee. Atomic cross-pod lock. Exactly one worker across any number of pods and any number of firing sources can hold this key at a time.
4
Idempotent downstream call
Pass job_id as the idempotency key to the webhook or microservice. If two calls somehow slip through layers 1–3, the receiver ignores the second. Belt and braces.

Dual-enqueue (Redis + SQS)

def schedule_job(job_id: str, delay_seconds: int):
    fire_at = time.time() + delay_seconds

    # Primary: Redis sorted set (unlimited delay, ~1s precision)
    redis.zadd("scheduled_jobs", {job_id: fire_at})

    # Fallback: SQS FIFO delay queue (resilience if Redis fails)
    if delay_seconds <= 900:  # SQS max = 15 min
        sqs.send_message(
            MessageBody=job_id,
            DelaySeconds=delay_seconds,
            MessageDeduplicationId=job_id,  # layer 1: queue-level dedup
            MessageGroupId="delayed-jobs"
        )

Redis sorted set poller

def poll_redis():
    while True:
        now = time.time()
        jobs = redis.zrangebyscore("scheduled_jobs", 0, now)
        for job_id in jobs:
            removed = redis.zrem("scheduled_jobs", job_id)
            if removed:  # layer 2: ZREM atomic claim
                executor.submit(process_job, job_id, "redis")
        time.sleep(1)

Worker with full dedup fence

def process_job(job_id: str, source: str):
    # Layer 3: atomic fence across all pods
    acquired = redis.set(
        f"dedup:{job_id}", 1,
        nx=True, ex=120
    )
    if not acquired:
        logger.info(f"[dedup] drop duplicate job={job_id} src={source}")
        if source == "sqs":
            sqs.delete_message(job_id)  # always ack — never re-queue
        return

    try:
        # Layer 4: idempotency key on downstream call
        webhook.call(
            job_id=job_id,
            idempotency_key=job_id  # receiver ignores duplicate calls
        )
        if source == "sqs":
            sqs.delete_message(job_id)
        delivery_log.write(
            job_id=job_id, source=source,
            delivered_at=time.time()
        )
    except Exception as e:
        # CRITICAL: release fence → job stays retry-eligible
        redis.delete(f"dedup:{job_id}")
        metrics.increment("job.failed", tags={"src": source})
        raise

SQS consumer (separate worker)

def consume_sqs():
    while True:
        msgs = sqs.receive_messages(
            MaxNumberOfMessages=10,
            VisibilityTimeout=90,   # must be < fence TTL
            WaitTimeSeconds=20        # long poll
        )
        for msg in msgs:
            executor.submit(process_job, msg.body, "sqs")

Quiz

Six questions covering everything in this session.

Q1. Why can't Kafka guarantee a message is delivered exactly 50 seconds after being produced?
Kafka doesn't support delayed messages in its API
Kafka is pull-based — consumers decide when to poll, so delivery timing is not guaranteed
Kafka uses TCP which adds variable latency
Kafka partitioning prevents time-based ordering
Correct. Kafka is a commit log that consumers pull from at their own pace. There is no mechanism in Kafka to say "deliver at t+50s". The delay guarantee must live in a dedicated scheduler (Redis sorted set, SQS, timing wheel).
Q2. You use Redis sorted set AND SQS as your delay queue. Both fire for the same job_id. What is the correct component to prevent duplicate execution?
Remove one of the two queues — redundancy causes duplicates
Use SQS FIFO dedup ID — it covers cross-queue deduplication
A Redis SET NX EX fence — an atomic third store that both paths check before executing
Set the job status to "processing" in PostgreSQL before executing
Correct. Neither queue owns deduplication — a third atomic store must arbitrate. Redis SET NX EX is atomic and cross-pod. SQS FIFO dedup only prevents re-enqueue within 5 minutes on the same queue. A database status update is not atomic without a transaction + lock.
Q3. Worker acquires the fence (SET NX succeeds), then the downstream webhook call fails with a 500. What must the worker do?
Let the fence expire naturally via its TTL — don't touch it
DEL the fence key before re-raising the exception
Extend the TTL and retry inline
Write the failure to DynamoDB and exit
Correct. If you don't DEL the fence on failure, the job is locked out of retry until the TTL expires. If TTL > retry window, the job dies silently. Always release the fence in the exception handler to keep the job retry-eligible.
Q4. Your fence TTL is 30s. Your SQS visibility timeout is 60s. What goes wrong?
Nothing — the shorter TTL means jobs process faster
SQS will refuse to deliver the message
The fence key expires at 30s, SQS redelivers at 60s, a second worker acquires the fence and executes the job again — double execution
Redis evicts the key at 30s and the job is lost
Correct. The fence TTL must always exceed the SQS visibility timeout. If the fence expires before SQS redelivers, the dedup guarantee breaks. Rule of thumb: TTL = max(SQS_visibility_timeout, poll_interval) + 20s buffer. For a 50s job with 60s visibility timeout → TTL = 120s.
Q5. A timing wheel has O(1) insert and fire. Why?
It uses a min-heap sorted by fire time
Jobs are inserted into a bucket by fire_time % wheel_size — no sorting needed. Firing drains the current bucket.
It pre-computes all future fire times at startup
It uses Redis ZADD which is O(log N)
Correct. A timing wheel is a ring buffer of buckets. Insert is bucket[time % size].append(job) — O(1). Fire is bucket[tick].drain() — O(K) where K is the number of jobs in that bucket. No heap, no sort. Used by Kafka's internal purgatory and Netty's HashedWheelTimer.
Q6. For WebSocket routing at 1B users, why is consistent hashing important?
It reduces Kafka partition count
WebSocket connections need SSL termination at a fixed IP
Each WebSocket server holds stateful connections. A message for user X must route to the server holding X's socket — not broadcast to all servers.
SQS requires a consistent hash for message deduplication
Correct. WebSocket connections are stateful — each server holds open sockets for a subset of users. If you broadcast every message to every server (N² fan-out), it doesn't scale. Consistent hashing routes user X's events to exactly the one server holding X's socket.
