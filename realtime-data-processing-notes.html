Section 02

Kafka — The Highway

Every event flows here before anything else touches it

What it is: A durable, ordered, append-only log. Every event published stays there for a configurable time (7 days default). Any consumer can read from any offset — independently, without blocking others.

Why not send directly to each service? Without Kafka, the score API would push to leaderboard + fraud + notifications + analytics simultaneously. One slow consumer blocks all. One crash loses the event. Kafka decouples producers from consumers completely.

Key mental model

Kafka is not a database. It's a durable ordered log — like a newspaper that never goes out of print. Consumers track which "page" (offset) they've read up to. They can re-read from any point in history without affecting other consumers.

RocksDB — Flink's Local Brain

The most important piece most people skip over

What is RocksDB? It's an embedded key-value database — like SQLite but built for high write throughput and SSDs. It was created by Facebook (forked from Google's LevelDB). "Embedded" means it runs inside the Flink process on each worker machine — there's no separate RocksDB server. No network hop. No serialization overhead. The Flink worker reads and writes to it like reading a local file.

Why not just use RAM (HashMap)? A pure in-memory HashMap cannot hold state for 2.3 crore teams — that'd require ~100GB+ RAM per worker. RocksDB uses RAM for hot data and spills to SSD for cold data automatically. You get near-memory speed for recently accessed keys and disk-scale capacity for the rest.

How RocksDB stores data internally
WRITE PATH — LSM Tree
1. Write → MemTable (RAM)
2. MemTable full → flush to SST file on SSD
3. Background compaction merges SST files
4. Result: fast writes, sorted on disk
READ PATH
1. Check MemTable (RAM) first — fastest
2. Check Block Cache (RAM) — fast
3. Check SST files (SSD) — slower
4. Bloom filters skip unnecessary disk reads
📸
Checkpointing to S3 — every 30 seconds, Flink snapshots all RocksDB state stores across all workers and uploads to S3/HDFS. If a worker crashes mid-match, it restarts, downloads the last checkpoint, and resumes from exactly that point. Zero data loss. This is what makes Flink production-safe — without it, a crash would lose all in-memory scores.
Why RocksDB and not Redis for Flink state? Redis is a separate server — every state read/write would be a network call. At millions of events/sec, that's catastrophic latency. RocksDB is in-process — the call cost is nanoseconds. Flink uses Redis only for the final output (top-10K leaderboard), never for internal computation.

Windowing — answering "last N minutes" questions

Tumbling Window

Fixed non-overlapping slices

Every 5 minutes: process and emit. Like turning pages — no overlap. Dream11: publish leaderboard snapshot to social every 5 min.

Sliding Window

Overlapping slices

Every 1 min, look back 10 min. More compute, smoother signal. Fraud: "did this user do 20+ actions in any 10-minute window?"

Session Window

Inactivity-based

Group events until user idle for 30 min. Dynamic size — could be 2 min or 2 hours. Dream11: session length and engagement depth.

Section 04

Apache Spark — The Factory

Massive data, heavy computation, no urgency for milliseconds

Core superpower: In-memory computation across hundreds of machines. Old Hadoop wrote intermediate data to disk after every step. Spark keeps it in RAM — 10–100× faster. Its Catalyst optimizer also builds a DAG of all transformations and optimises the whole plan before executing (same idea as a SQL query planner).

⚙️ Apache Spark — when to use

📦Data is already sitting somewhere (S3, warehouse)
🔗Need joins across multiple large tables
🤖Training ML models (need multiple passes over all data)
⏱️Result needed in minutes or hours — not milliseconds
📊ETL, reporting, feature engineering

⚡ Apache Flink — when to use

🌊Events are streaming in right now via Kafka
⚡Need to react within milliseconds
💾Need stateful logic across events (counts, sums, heaps)
🚨Fraud detection, live score updates, real-time alerts
🔄Always-on job — not triggered, always running
🤖
ML Model Training
Dream11 / Netflix
Train player recommendation on 90 days of match + user data. Runs nightly at 2 AM. Output: model file on S3. Netflix: retrain "next watch" on 200M user histories. Takes 45 min — nobody cares, runs while you sleep.
🔄
ETL Pipelines
Uber / Swiggy
Raw ride_started + payment_captured events → joined clean trips table in warehouse. Spark joins messy sources from 10 microservices into one clean record. Runs hourly.
🕵️
Batch Fraud Analysis
PayPal / Razorpay
Graph analysis on billions of transaction edges. Who sent money to flagged accounts 3 hops away? Patterns only visible across weeks. GraphX (Spark) runs nightly, outputs risk scores per account by morning.
📊
Business Reporting
Flipkart / Dream11
Daily GMV by category, region, seller → CEO dashboard. Post-match: which players were picked most, win rates by user tier. Spark computes over the full day's data in minutes.
🧮
Feature Engineering
Zomato / Swiggy
"Avg order value per user per cuisine" across 2 years. 200 features × 10 crore users. Pre-computed daily by Spark → written to feature store → ML model reads at inference time. Never computed live.
📋
Log Analysis
Any large tech co.
Scan 10TB of yesterday's server logs. Find P99 latency spikes by service, error rate by API. Incident post-mortems. Capacity planning. Spark processes it in minutes across a cluster.
// Latency decision — which processing tool to pick
latency needed < 100ms → Flink // fraud, live scores, alerts
latency needed 1–30s → Spark Structured Streaming // supply/demand balancing
latency needed minutes → Spark batch // ETL, feature engineering
latency needed hours → Spark batch // ML training, graph analysis
Spark Structured Streaming — Spark also reads from Kafka in micro-batches every 1–5s. Not Flink-level real time, but good enough for many use cases. Swiggy uses it for delivery partner supply-demand: every 10s, count available partners vs active orders per zone and adjust fee estimate.

Cassandra schema — append-only design

-- WRONG mental model: UPDATE scores SET pts=54 WHERE teamId=77
-- Cassandra uses LSM tree — every UPDATE is secretly an INSERT anyway
-- So design append-only from the start ↓

CREATE TABLE team_scores (
  match_id TEXT,    -- partition key part 1
  team_id TEXT,    -- partition key part 2
  over TEXT,    -- clustering key — rows sorted within partition
  pts_delta INT,     -- how much changed THIS ball: +6, 0, +4
  cumulative INT,    -- running total: 48→54→54→58
  rank_at INT,     -- rank at this over snapshot
  PRIMARY KEY ((match_id, team_id), over)
);

-- What 3 balls look like as rows (all INSERTs, never UPDATE)
-- match_id | team_id | over | pts_delta | cumulative | rank_at
-- M101 | 77 | 15.1 | +6 | 54 | 1
-- M101 | 77 | 15.2 | 0 | 54 | 1
-- M101 | 77 | 15.3 | +4 | 58 | 1

-- Query: team_77's full history → one partition scan, instant
SELECT over, cumulative, rank_at FROM team_scores
WHERE match_id='M101' AND team_id='77';

Step 2 — Replay 3 deltas

Step 3 — Merge + top-5 sort in Druid. ORDER BY total_pts DESC LIMIT 5. Result in <1 second — even over 27 crore rows total.
Section 06

Storage Strategy — Redis vs Druid vs Elasticsearch

Same data, completely different jobs. Never one DB for all.

Redis vs Druid

Redis sorted set

⚡Latency: <1ms always
🎯Query: ZRANK teamId → my rank right now
📸Data: current state only. No history.
💸Cost: expensive RAM. ~100M entries practical max.
✅Use for: live app reads per-user during match

Apache Druid

🕐Latency: 100ms–2 seconds
📊Query: SQL top-N across billions of rows
📚Data: all history, every over, queryable
💰Cost: columnar compression — billions of rows fine
✅Use for: analytics, historical rank queries, dashboards
Why Redis still exists alongside Druid

Druid cannot give sub-millisecond latency. Even a fast Druid query is 100ms+. At 5 crore users refreshing every few seconds: 100ms × 5 crore concurrent requests = you need a thousand Druid nodes. Redis at <1ms handles the same load with a handful of machines. Redis is the scoreboard on the wall. Druid is the statistician in the back office.

Druid vs Elasticsearch — internal design difference

Druid — columnar store

All pts values for all rows sit physically adjacent on disk. SUM(pts) reads only that column — never touches teamId, city, device. Pre-computes rollups at ingest time.

✓ Great: aggregations, top-N, time-series, COUNT/SUM/AVG over billions

✗ Bad: full-text search, frequent updates

Elasticsearch — inverted index

For every word in your documents, stores a list of which docs contain it. "NullPointerException" → [doc3, doc9, doc41]. Word lookup = O(1).

✓ Great: full-text search, log analysis, Kibana dashboards

✗ Bad: aggregating crore-scale numerical data

Druid vs Postgres — not the same thing at all

DimensionPostgresDruid
StorageRow store — full row on disk togetherColumn store — each column stored separately
UPDATEFull in-place update (MVCC)Append-only. No UPDATE or DELETE.
TimeJust another columnFirst-class. Auto-partitions by time.
Pre-aggregationNoYes — rollups computed at ingest
Best forTransactions, joins, ACIDTime-series analytics, billions of rows
SUM across 2.3 crore rowsSlow — reads all columns firstFast — reads only pts column, in memory

// Decision tree — run through this in any interview if query == "find documents matching text / search logs": return Elasticsearch // log search, Kibana, full-text elif query == "my rank right now" and latency < 1ms: return Redis // sorted set ZRANK elif query == "top-N / SUM / COUNT over time window": return Druid // columnar OLAP, pre-aggregated elif query == "one entity's full history in time order": return Cassandra // partition by entity, cluster by time elif query == "complex joins + ACID transactions": return Postgres // financial records, user accounts elif query == "Flink internal state — zero latency": return RocksDB // embedded in Flink worker, in-process
Section 08

Interview One-Liners

10 questions they will definitely ask — crisp answers ready

QuestionAnswer
Why Kafka before Flink?Decouples producers from consumers. Multiple systems read independently. Kafka is a durable buffer — if Flink restarts, it replays from last committed offset. No event loss.
Why not update all 2.3 crore in Redis per ball?Only ~10K teams in Redis (hot tier). Flink maintains a min-heap in state. Only boundary-crossing teams trigger Redis writes — 2–5 ZADD ops per ball, pipelined in one network call.
What does Flink keep in RocksDB state?Three things: player score map, team cumulative score map, and min-heap of top-K teams. All local to the worker — no external DB queried per ball. That's why it's fast.
Why RocksDB inside Flink and not Redis?RocksDB is in-process — no network hop. Redis is a separate server — every state read/write adds network latency. At millions of events/sec that's fatal. Redis is only used for the final output.
What if Flink crashes mid-match?Checkpointing: Flink snapshots all RocksDB state to S3 every ~30s. On restart, loads last checkpoint and resumes. Zero data loss by design.
Why can't Cassandra do top-N across all teams?Partition key is (matchId, teamId). Top-N = 2.3 crore partition scans. Takes minutes. Druid is the right tool — columnar, pre-indexed, one-column scan for aggregations.
How do you answer "top 5 at over 15.3"?Nearest snapshot (over_15.0, pre-indexed in Druid) + replay 3 delta balls from Cassandra. Merge and top-5 sort in Druid. Under 1 second.
Druid vs Postgres?Druid: columnar, append-only, time-series, pre-aggregation. Never use for transactions or updates. Postgres: row store, ACID, UPDATE/DELETE, joins. Never use for billion-row analytics scans.
Why not just Druid for everything?Druid has 100ms+ latency. Live app needs <1ms for rank reads during match. Redis wins on latency. Druid wins on scale and analytical depth. They solve different problems.
When Spark vs Flink?Flink: always-on, millisecond reactions, stateful per-event logic. Spark: kicked off on schedule, minutes latency, heavy batch — ML training, ETL, reports, graph analysis.

The senior engineer signal

The sign of a senior engineer is knowing which tool's weakness disqualifies it for a given job — not just what each tool does. Redis is disqualified for historical analytics (no history). Cassandra is disqualified for top-N across all teams (wrong partition model). Druid is disqualified for live reads (too slow). Always reason from access pattern first, then pick the tool.
