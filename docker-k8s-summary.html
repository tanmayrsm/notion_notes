Section 01

Cluster Architecture — Docker → Pod → Node → Cluster

Docker is just the packaging format — the "box" your app lives in. A Pod wraps one or more containers that share a network and disk. Nodes are the actual servers that run pods. The Cluster is the whole environment orchestrated by Kubernetes. Vault runs as a sidecar container inside the same pod as your app, injecting secrets transparently.

Docker container
The packaged application — your Spring Boot JAR, its runtime, dependencies, all bundled into one portable image. Runs identically on any machine.
Pod
The smallest deployable unit in Kubernetes. Wraps one or more containers that share localhost networking and mounted volumes. Usually one app container + optional sidecars.
Node
An actual server — VM or physical — that runs pods. Kubernetes can span hundreds of nodes. The kubelet agent on each node reports to the control plane.
Vault sidecar
Runs alongside your app container in the same pod. Fetches secrets from the central Vault server and writes them to a shared in-memory volume. Your app never touches Vault directly.
Section 02

API Gateway — Where It Lives & How Traffic Flows

The API Gateway is not a special dedicated server — it is just another pod inside your cluster, exposed to the outside world via a LoadBalancer service. Downstream Spring Boot services use ClusterIP — invisible to the internet, reachable only inside the cluster by DNS name.

How Vault secret injection works step by step:

Step 01
Pod starts
Kubernetes creates the pod. The Vault agent sidecar container launches alongside your Spring Boot container.
Step 02
Agent authenticates
Vault agent uses the pod's Kubernetes Service Account token to prove its identity to the central Vault server.
Step 03
Secrets fetched
Agent retrieves DB passwords, API keys, TLS certs from Vault and writes them to a shared in-memory volume inside the pod.
Step 04
App reads config
Spring Boot reads secrets from that volume as environment variables or files. It never calls Vault directly — zero hardcoded credentials.
Section 03

kubectl in Real Production — What Teams Actually Run

kubectl is far more than a log viewer. It's the command-line remote control for everything in the cluster — used during deployments, incidents, debugging, scaling, config changes, and node maintenance. For JVM heap / GC monitoring specifically, kubectl top pods gives OS-level memory, and kubectl exec lets you run jcmd / jmap inside the container.

Incident
First thing to run when prod breaks
Check pod state, describe to see events, get previous logs from before the crash restart.
kubectl get pods -n production

kubectl describe pod <name>

kubectl logs <pod> --previous
--previous is crucial — gets logs from the crashed version, not the freshly restarted one.
Deploy
Push new image, watch rollout, instant rollback
Zero-downtime rolling updates built in. One command to undo.
kubectl set image deployment/svc app=v2.1

kubectl rollout status deployment/svc

kubectl rollout undo deployment/svc
Undo rolls back to the exact previous ReplicaSet. Kubernetes keeps a revision history.
Debug
Shell into container — like SSH
When logs aren't enough — check env vars, run curl, inspect the filesystem.
kubectl exec -it <pod> -- /bin/sh

kubectl exec <pod> -- env | grep DB

kubectl exec <pod> -- curl localhost:8080/health
Also: run jcmd / jmap here for JVM heap dumps and GC analysis.
Scaling
Manual scale + inspect autoscaler
Traffic spike? Scale immediately. HPA handles automatic scaling by CPU/memory threshold.
kubectl scale deployment/svc --replicas=10

kubectl get hpa
HPA = Horizontal Pod Autoscaler. kubectl lets you inspect or override its current state.
Memory / GC
Live resource usage per pod and node
OS-level view of memory and CPU consumption across the cluster.
kubectl top pods -n production

kubectl top nodes
Combine with Prometheus + Grafana for JVM GC pause times, old-gen pressure, and heap charts.
Node maintenance
Drain and cordon for patching
Drain gracefully moves pods off before maintenance. Cordon stops new pods landing on a node.
kubectl drain <node> --ignore-daemonsets

kubectl cordon <node>
Always drain before patching or rebooting a node. Workloads reschedule to healthy nodes automatically.
