1. What GC does & why it's hard

Every time your Java code runs new SomeObject(), the JVM puts that object in a region of memory called the heap. When nothing in your program holds a reference to an object anymore, it's dead weight — but it still sits in memory until GC removes it.

⚠️

The core challenge: GC needs to inspect the entire heap to find garbage — but your app is simultaneously creating and discarding objects. Stopping the app to inspect is safe but slow. Running concurrently is fast but requires careful synchronization. Every GC design is a different trade-off on this dial.

2. How mark & sweep works

This is the foundation algorithm under every modern GC. Two phases:

🌱
Start from roots
Stack vars, static fields
🔗
Follow references
Graph traversal
✓
Mark reachable
Everything reached = live
🗑
Sweep unreachable
Unmarked = deleted

Tri-color marking (how concurrent GC stays safe)

When GC marks while your app runs, it uses three colors to track progress:

ColorMeaningAction
White Not yet visited — possibly garbage Candidates for deletion at sweep time
Gray Found, but references not yet fully scanned In the work queue — GC will process next
Black Fully scanned — definitely alive Safe — all its references are accounted for
🧠

The invariant that must never break: A black object must never directly point to a white object. If it does, GC would delete the white object even though it's reachable. The write barrier (section 5) is what enforces this invariant during concurrent marking.

3. Generational GC

The key empirical observation: most objects die young. A String built to format a log line, a request wrapper, a DTO — all created and discarded in milliseconds. GC exploits this by separating the heap into generations:

Young generation

  • New objects land here in Eden
  • Two small Survivor spaces (S0, S1)
  • Collected frequently — Minor GC
  • Most objects never leave — they die here
  • Fast to scan because it's small

Old generation

  • Objects that survive ~15 minor GCs get promoted here
  • Collected infrequently — Major / Full GC
  • Expensive to scan — it's large
  • Filling this up is the main path to Full GC
  • Target: keep old-gen occupancy low
✅

The generational hypothesis in one line: Minor GC is cheap because almost everything in Eden is already dead by the time GC runs. You're scanning a tiny space and finding very few survivors.

4. G1 GC — heap regions & remembered sets

Why G1 exists

Traditional GC had to collect entire regions at once — pause time was proportional to heap size. G1 breaks the heap into small equal-sized regions (1–32 MB each) so it can collect any subset and keep pauses bounded.

📦

G1 = "Garbage First" — it ranks regions by how much garbage they contain and collects the most garbage-dense ones first. That's the name, that's the whole strategy.

The heap as a region grid

Remembered Sets (RSets) — the key data structure

If G1 only collects some regions, how does it know all the objects pointing into those regions? Without that, it could miss live objects and corrupt the heap.

Each region has its own RSet — a list of (other-region, card) pairs that point INTO it. During GC, instead of scanning the whole heap, G1 only needs to scan roots + the RSets of collected regions.

💸

RSet cost: RSets consume real memory — typically 1–5% of heap. In write-heavy workloads (updating many references), maintaining RSets costs CPU too. This is the hidden price of G1's flexibility.

How the pause budget works

StepWhat happens
1. Set pause target-XX:MaxGCPauseMillis=200 — G1 will try to stay under 200ms
2. Rank regionsSort old-gen regions by reclaimable bytes ÷ estimated scan time
3. Fill the budgetGreedily pick top regions until estimated pause time is consumed
4. EvacuateSTW: copy live objects from selected regions into free regions
5. ReleaseOld CSet regions are now entirely free — recycled for future use

5. Advanced internals

TLAB — why object creation is nearly free

Each thread has a private chunk of Eden called a Thread-Local Allocation Buffer. Allocating inside a TLAB is just moving a pointer one step — no locks, no synchronization with other threads.

TLAB full — what happens

  1. Current TLAB is retired (waste filled with dummy object)
  2. Thread gets a fresh TLAB from Eden
  3. If Eden itself is full → Minor GC fires
  4. Very large objects skip TLAB → go directly to old gen (danger!)

TLAB tuning flags

  • -XX:TLABSize — override default TLAB size
  • -XX:TLABWasteTargetPercent — max % waste before retirement
  • Usually let JVM auto-size — it learns from refill patterns
  • High refill rate in JFR → mixed small + large allocations

Write barrier — keeping concurrent marking correct

Problem: your app mutates references while GC is mid-marking. A black object could get a new reference to a white (unscanned) object — violating the tri-color invariant and causing GC to delete a live object.

SATB (Snapshot-At-The-Beginning) — G1's solution. When you overwrite a reference, the old value is saved and treated as gray. GC might retain some garbage until next cycle (floating garbage), but it never deletes live objects.

Safepoints — the hidden latency source

Before GC can run, every app thread must reach a safepoint — a specific safe pause location in its execution. GC can't pause threads mid-instruction. This means:

🚨

Time-to-safepoint latency: If Thread A has a long loop with no safepoint poll inside, GC must wait for that loop to finish before starting. Your total pause = GC work time + time waiting for the slowest thread. This is invisible in most GC logs and is a common source of mysterious long pauses.

CauseFix
Long counted loops (no safepoint poll)-XX:+UseCountedLoopSafepoints (JDK 14+)
Biased lock revocation (pre JDK 15)Upgrade to JDK 15+ (biased locking removed)
jmap -dump called on live processUse jcmd GC.heap_dump instead; drain traffic first
Class redefinition (hot-reload)Avoid in production; only in dev environments

Why ZGC achieves sub-millisecond pauses

G1 must stop your app to move (evacuate) objects. ZGC moves them while your app runs using two tricks:

  1. Load barriers: Every time your code reads an object reference, a tiny piece of JIT-inserted code checks if that object has been moved. If yes, it transparently updates the pointer before returning it. Your app never sees stale addresses.
  2. Colored pointers: ZGC stores GC metadata (marked, remapped) directly in spare bits of the 64-bit pointer. This lets it check object state with a single pointer read, with no separate metadata lookup.

8. Commands reference

Build-time flags — paste into your service config

# Recommended baseline for a latency-sensitive 1B rpm service

-XX:+UseZGC                              # ZGC for sub-ms pauses (JDK 17+)
-Xms16g -Xmx16g                        # Lock heap — prevent resize pauses
-XX:+AlwaysPreTouch                      # Pre-fault pages at startup, not mid-traffic
-Xlog:gc*:file=gc.log:time:filecount=5,filesize=20m
-XX:+HeapDumpOnOutOfMemoryError          # Auto-dump on OOM for post-mortem
-XX:HeapDumpPath=/tmp/oom.hprof
-XX:+ExitOnOutOfMemoryError              # Fail fast — let Kubernetes restart cleanly
-XX:+UseCountedLoopSafepoints            # Reduce time-to-safepoint latency
-XX:+FlightRecorder                      # JFR always available for on-demand profiling

# G1-specific additions (if using G1 instead of ZGC):
-XX:MaxGCPauseMillis=200
-XX:InitiatingHeapOccupancyPercent=35   # Start concurrent marking earlier
-XX:G1HeapRegionSize=16m               # Tune for your object size distribution

Runtime detection — diagnosing a live problem

See GC happening right now

# Real-time GC stats, refresh every 1 second
# Replace 12345 with your PID (find it: jps -l)
jstat -gcutil 12345 1000

Emergency — free memory on production

⚠️

Honest warning: There is no free-memory-instantly command without trade-offs. Every option below causes a pause or risks instability. The real answer is to drain the pod and restart it.

Option 1 — trigger GC manually (fast, temporary)

# Via jcmd — request a full GC on a running process
jcmd 12345 GC.run

# This WILL cause a pause. At 1B rpm, drain traffic off this pod first.

Option 2 — take heap dump then restart (safest)

# 1. Remove pod from load balancer rotation
# 2. Capture heap for post-mortem
jcmd 12345 GC.heap_dump /tmp/emergency.hprof

# 3. Restart the pod (Kubernetes handles this cleanly)
# 4. Analyze dump offline with Eclipse MAT or VisualVM

Option 3 — clear a cache that's hoarding memory

# Via an admin/actuator endpoint — evict all entries
cache.invalidateAll();

# Or reduce max size temporarily then restore
cache.policy().eviction().ifPresent(e -> e.setMaximum(0));

The 3-step incident response

1
Triage (30s)
jstat — is FGC rising? Old gen >90%? Confirm it's GC.
2
Contain (2 min)
Drain pod. jcmd GC.run to buy time. Heap dump before restart.
3
Fix (post-incident)
async-profiler → find root cause → fix code, not just flags.

9. What to say in the interview — 1B rpm

🎯

The test: At 1B requests per minute, the interviewer is not asking "what flags do you set." They're asking: can you reason about GC at system scale and connect low-level JVM behavior to user-facing impact?

Opening — set the stakes

Say this first
"At 1 billion requests per minute, we're allocating 50–200 GB of objects per second. GC isn't background noise — it's on the critical path of every request. A 200ms GC pause at that scale means millions of requests either queue up or time out. So the design decision isn't just which collector to use, it's how to architect the service so GC pressure is minimal in the first place."

Collector choice

When asked G1 vs ZGC
"I'd default to ZGC on JDK 17+. G1 is excellent up to ~32GB heaps but at 1B rpm the pressure is extreme. The key difference: ZGC moves objects concurrently while requests are being served — pauses stay under a millisecond. G1 has to stop the world to evacuate objects, so pause time scales with how much it needs to move."

Code-level design

When asked what you'd actually change
"I'd go after allocation rate before touching JVM flags. Profile with async-profiler in allocation mode — usually 2–3 hotspots account for 80% of allocations. Common culprits: boxing primitives in maps, building intermediate strings in hot paths, stream chains creating wrapper objects. For network payloads I'd use DirectByteBuffer to keep large buffers off-heap entirely. The goal is to make GC boring — infrequent minor collections, almost zero old-gen pressure."

Failure scenario — if they push harder

When asked about worst-case failure
"The failure mode I'd watch for is evacuation failure — when G1 can't find free regions to copy live objects into. That triggers a full stop-the-world collection that can last seconds. I'd prevent it by sizing the heap with 40–50% headroom, setting InitiatingHeapOccupancyPercent to 35 so concurrent marking starts well before old gen fills, and watching FGC count in jstat as a real-time signal."

Instant questions — what to say without hesitation

Interviewer asksYour answer (one line)
"What's a safepoint?"A pause checkpoint every thread must reach before GC can start. Slow threads increase time-to-safepoint, adding hidden latency beyond the GC itself.
"What's a write barrier?"JIT-injected code on every reference write that keeps concurrent marking correct — it logs overwritten references so GC can re-scan them.
"How does TLAB help?"Each thread allocates in its own private Eden chunk — no locks, just pointer bump. Makes object creation nearly free.
"What's SATB?"G1's technique: snapshot all live references at marking start, log overwrites, re-scan them. May retain some floating garbage but never deletes live objects.
"How would you free memory in prod?"Drain the pod, run jcmd GC.run to buy seconds, take a heap dump, restart. Analyze dump offline. Fix root cause in code.
"What command shows live GC?"jstat -gcutil [PID] 1000 — updates every second, shows Eden %, Old gen %, full GC count.
❌

Never say: "I'd just increase heap size and use ZGC." That signals you treat GC as a black box. Always connect flags to root causes and system behavior.

✅

The answer that wins the room: "I'd fix the allocation hotspot in code. Flags buy you time. Code changes save you permanently. GC tuning should be the last resort, not the first."
